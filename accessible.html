Part 1: Input

Interaction Processes & Differences

Across the three applications—Safari Browser History, Zotero, and Visual Studio Code—mouse usage generally allows basic selection, opening items with clicks or double clicks, and right-click for additional options. However, Safari’s history list requires a double click (rather than a single click) to open a page, which feels unintuitive. Zotero enables double-click editing but doesn’t allow multi-select via mouse. In VS Code, mouse interactions can include drag-and-drop in the file explorer, but no shift-based multi-select in certain views.

Functionalities & Usability Observations

With the keyboard, Safari supports arrow-key navigation and multi-select with Shift, but it doesn’t provide access to “Clear History” or open-in-new-tab functionality. Zotero’s keyboard approach lets users edit text fields immediately on focus, but again lacks multi-select. VS Code offers extensive keyboard shortcuts, though Tab can be overridden by code indentation, and toggling between editing mode and navigation can be tricky. Touch interactions are only partially available: Safari offers a long press for context options, while Zotero and VS Code lack dedicated mobile versions, effectively removing multi-select or editing features on touch devices.

Learnability, Memorability, Efficiency, and Mobile Considerations

Each component’s learnability depends on how familiar users are with standard interaction patterns: Safari’s double-click requirement is surprising, Zotero’s auto-entry into edit mode can be unexpected, and VS Code’s Tab indentation may confuse users expecting typical UI navigation. Over time, these behaviors become memorable, but they’re less efficient initially. For applications without mobile/touch support (Zotero, VS Code), essential functionalities (editing, multi-select, or drag-and-drop) do not translate to a mobile experience. This limits efficiency and learnability for users accustomed to the desktop versions, potentially causing frustration when switching to mobile or tablet usage where those interactions are unavailable.

Part 3: State Models & Redesign

My redesign of VS Code's exploratory panel, which enhances user's experience in exploring nested accordion structures. Previously, users had to "expand" or "collapse" individual accordions, which is time-consuming and can be confusing for users with visual impairments. For example, they may expand a few nested accordions only to notice this is not the branch they intended to explore. Thus, they would have to manually collapse each of them before expanding the next branch, hoping it it intended. In the redesign, I introduce a "show all" and "hide all" buttons that allow users to expand or collapse all accordions at once, providing a more efficient way to navigate complex nested structures. The tradeoff here is information overload. If all nested accordion options are displayed at the same time, it could be too much to be comprehendible at once, especially for expert users who are very familiar with the directory structure. However, this can be mitigated by providing a clear visual hierarchy and grouping of the nested accordions.

Part 4: Reflection

In Safari, Zotero, and VS Code, many basic UI features work well for usability and accessibility—such as clear visual focus states, arrow-key navigation, and highlighting selected entries. However, they sometimes miss deeper accessibility elements like announcing depth in nested structures. My redesign in the VS Code Explorer Panel adds explicit depth indicators, reducing guesswork for everyone and providing critical clarity for screen readers and users with limited spatial memory.

Some accessibility considerations—like ensuring the visualization of depth of nested features can be easily interpreted by people with visual impairments—aren’t always addressed in these applications. In my redesign, I account for this by announcing depth levels audibly and visually, solving a “mismatch” between hierarchical navigation and a user’s ability to track complex nesting (as described in Kat Holmes’ framework).

  • Keyboard-Only: If depth isn’t announced, users navigating complex folders can get lost. Adding verbal depth cues reduces cognitive strain.
  • Mouse & Motor Impairments: Reliance on hovering or drag-and-drop can exclude users with limited precision; adding explicit click-based navigation ensures more inclusive access.

Historically, mouse and keyboard interactions are most commonly prioritized in component design. This can marginalize touch users and screen-reader users who rely on well-defined focus and spoken feedback, ultimately impacting learnability and efficiency for those groups. By designing with screen-reader and touch inputs in mind from the start, we create more inclusive, intuitive experiences for all.
