Accordion Accessibility Analysis

Inputs, Outputs, State Models, & Inclusive Redesign

Part 1: Input

Describe how the accordion is used with mouse, keyboard, and touch across three applications (e.g., Safari, Zotero, VS Code). Include observations such as:

  • Safari: Mouse interactions include right-click options and double-clicking entries; keyboard navigation via arrow keys, but limited access to "Clear History"; touch interactions use long-press for options.
  • Zotero: Double-click editing for text; keyboard tabbing auto-enters editing mode, but no multi-select; no mobile version available.
  • VS Code: Drag-and-drop in file explorer; keyboard interactions involve entering edit mode with Enter, but Tab sometimes adds indentation; no touch support.

Part 2: Output

Discuss the visual and non-visual outputs of the accordion component. Consider:

  • Visual cues (e.g., ‘>’ for collapsed and ‘⌄’ for expanded states) with color highlighting.
  • Screen reader outputs that announce state, position, and focus order.
  • Differences in how relative position and nested depth are communicated (or not communicated) across applications.

Part 3: State Models & Redesign

Present your state diagrams for mouse and keyboard users. Explain how your redesign addresses the following:

  • Adding explicit depth indicators to improve spatial awareness.
  • Enhancing keyboard navigation with clear focus and toggle actions.
  • Balancing efficiency, learnability, memorability, and accessibility.

Part 4: Reflection

Reflect on the inclusive design principles applied to your redesign:

  • What worked well in terms of usability and accessibility in the observed components?
  • Which accessibility considerations were previously overlooked and how have you addressed them?
  • How does your change address a mismatch in current design practices?
  • Examples:
    • Without depth indicators, users must remember navigation steps, which can be challenging for those with visual impairments.
    • Prioritizing mouse and keyboard interactions often excludes touch and screen reader users, impacting overall user experience.
